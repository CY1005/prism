# Prism v3 原型生成提示词

> 创建时间：2026-04-09
> 基于：stitch-prompt.md (v2) 修改
> 核心变更：从"AI云平台竞品分析工具"升级为"通用项目知识棱镜"，支持多项目类型

---

我要设计一个完整的产品——Prism，项目知识管理与分析平台。它能收纳不同类型的项目知识（产品竞品分析、系统架构项目、研究平台项目等），并提供竞品分析、用户视角分析、产品合理性审计、需求影响范围分析和测试点生成能力。请严格按照以下描述生成所有页面，不要添加任何我没有描述的功能或元素。

## 全局风格要求
- 风格：极简、高效、干净。参考 Linear / Notion 的视觉风格
- 配色：黑白灰为主，仅用一个强调色（蓝色）标识关键操作和选中状态。完善度用绿/黄/红三色，除此之外不要多余色彩
- 无装饰性元素，无渐变背景，无阴影过重的卡片
- 字体层级清晰（标题/正文/辅助文字三级），行间距舒适
- 大量留白，信息密度克制，一屏不要塞太多内容
- 边框用细线或无边框，用间距区分区块
- 所有页面保持统一的导航栏、配色、间距、圆角风格
- 适合桌面端（1440px）
- 所有文字使用中文

## 全局导航栏（所有页面顶部固定）
- 左侧：Prism logo + 产品名
- 中间：全局搜索框（占位文字"跨项目搜索功能、模块、知识..."）
- 右侧：通知图标 + 用户头像 + 用户名"陈玥"

## 页面1：登录页
- 居中登录卡片，背景简洁
- 卡片内容：Prism logo + "欢迎回来" + 邮箱输入框 + 密码输入框 + 登录按钮
- 底部：没有账号？注册
- 不要第三方登录按钮

## 页面2：项目列表页（登录后首页）
- 顶部：面包屑"我的项目" + 右侧"新建项目"按钮
- 主体：卡片网格（2行2列）
- 每个卡片内容：
  - 左上：项目类型标签（小Badge，如"产品分析""系统架构""研究平台"）
  - 项目名称（大字）
  - 一句话描述
  - 三个数字指标（根据项目类型不同标签不同）：
    - 产品分析类：产品线数 / 模块数 / 完善度%
    - 系统架构类：组件数 / 功能数 / 完善度%
    - 研究平台类：模块数 / 功能数 / 完善度%
  - 最近更新时间
  - 底部：成员头像叠放
- 示例4个项目：
  - "AI云平台竞品分析" — 标签[产品分析] — "系统性分析AI云平台行业竞品设计与技术" — 3条产品线 / 42个模块 / 58%
  - "OpenClaw" — 标签[系统架构] — "个人AI陪伴系统，Telegram+Claude全栈" — 4个组件 / 15个功能 / 45%
  - "MappingStudio" — 标签[研究平台] — "AI驱动的科技公司研究报告平台" — 5个模块 / 23个功能 / 30%
  - "Prism" — 标签[自定义] — "用Prism分析Prism，dogfooding验证" — 3个阶段 / 12个模块 / 20%

## 页面3：新建项目弹窗
- 模态弹窗，覆盖在项目列表页上
- 上半部：项目名称输入框 + 项目描述输入框
- 下半部"选择模板"区域：4个模板卡片横排，每个卡片：
  - 模板图标 + 模板名称
  - 一行描述
  - 预设维度列表（小标签排列）
  - 点击选中后有蓝色边框高亮
- 4个模板：
  - 产品竞品分析：适合分析竞品产品设计与技术。预设维度：功能描述、用户场景、技术实现、设计决策、工程经验、测试分析、需求分析、竞品参考。层级：产品线→模块→功能项
  - 系统架构项目：适合记录系统设计与演进。预设维度：功能描述、接口规范、设计决策、工程经验、部署配置、测试分析。层级：系统层→组件→功能
  - 研究平台项目：适合数据产品和研究工具。预设维度：功能描述、用户场景、设计决策、工程经验、质量指标、成本分析、测试分析。层级：应用层→模块→功能
  - 自定义：自选维度组合和层级名称
- 底部：取消 + 创建按钮

## 页面7：功能档案页——系统架构类（OpenClaw示例）
- 左侧边栏：
  - 顶部：项目名"OpenClaw" + 项目类型标签[系统架构] + 折叠按钮
  - 树形导航，层级标签用"系统层→组件→功能"：
    - 📁 消息层

        - 📄 消息收发 ← 选中高亮

    - 📁 路由层

        - 📄 意图识别
    - 📁 技能层
    - 📁 记忆层

- 右侧主区域：
  - 顶部：面包屑"消息层 > Telegram Bridge > 消息收发" + 完善度"4/6 维度已填写 67%"
  
  - 注意：这里只有6个维度卡片（系统架构模板的维度集，不是8个）：

  - 卡片1 功能描述（展开）："通过Telegram Bot API的getUpdates长轮询接收用户消息，经消息队列转发至路由层处理后，通过sendMessage返回响应。"
  - 卡片2 接口规范（展开，这是产品分析模板没有的维度）：
    表格展示：
    | 方向 | 协议 | 端点 | 说明 |
    | 入站 | HTTPS | Telegram Bot API | getUpdates轮询 |
    | 出站 | HTTP | localhost:8900/webhook | 转发至Router |
  - 卡片3 设计决策（展开）：
    背景："Webhook vs 长轮询"
    决策："选择长轮询（getUpdates）"
    放弃："Webhook需要公网HTTPS域名，当前服务器不具备"
    后果："延迟略高（1-2秒），但部署简单"
  - 卡片4 工程经验（展开）：

    标签：[踩坑] [API限流] [Telegram]
  - 卡片5 部署配置（展开，这也是产品分析模板没有的）：
    "systemd服务：telegram-polling.service，开机自启，异常3秒后自动重启。依赖：message-router.service。"
    标签：[systemd] [自动重启]
  - 卡片6 测试分析（折叠）：摘要"已记录1个测试场景"

  - 底部：版本时间线（持续模式，显示日期而非版本号）：
    - 2026-04-07（蓝色圆点，"最新"）— "修复cron报错，升级router v2"

## 页面间跳转关系
- 登录页 → 登录成功 → 项目列表页
- 项目列表页 → 点击项目卡片 → 对应类型的项目全景图
- 项目列表页 → 点击新建项目 → 新建项目弹窗（含模板选择）
- 项目全景图 → 点击节点 → 对应类型的功能档案页
- 功能档案页 → 左侧树切换 → 更新右侧内容（维度卡片数量和类型根据项目模板变化）
- 全局搜索 → 回车 → 搜索结果页（支持跨项目）
- 搜索结果页 → 点击某条 → 对应项目的功能档案页
- 导航栏用户头像 → 下拉菜单 → 项目设置 / 运维后台 / 退出登录

## 严格要求
- 只生成以上描述的页面和元素，不要自行添加任何功能
- 所有页面风格必须统一
- 所有文字使用中文
- 不同项目类型的档案页维度卡片数量不同（产品分析8个、系统架构6个、研究平台7个），这是核心差异，必须体现
- 新建项目弹窗的模板选择必须展示每个模板的预设维度标签
