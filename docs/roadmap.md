## 开发方法论

### 每个功能的开发循环（IDAKE 五步）

```
1. Spec    → 写 AC（3-10 条），定义"做对了长什么样"
2. Build   → Claude Code 基于 AC 生成代码
3. Break   → CY 基于 AC 逐条验证，找边界 case
4. Fix     → AI 修复 → CY 再验 → 循环直到所有 AC 通过
5. Evolve  → 踩坑经验录入 Prism 自身
```

### Harness 机制

```
每个 session：
  开始 → 读 harness-progress.md，10 秒恢复上下文
  结束 → 更新 harness-progress.md（已完成/进行中/卡点/下一步）

自动检查：
  同一文件编辑 >6 次 → 停下来换思路
  同一 bug 修 >3 轮 → 记 pain-log，考虑绕过
  每版本结束 → 架构检查（目录结构/依赖方向/schema migration）
```

### 开发原则

1. **Spec 先行** — 不写 AC 不开始写代码
2. **按功能纵切** — 一个功能从 DB 到前端做完再做下一个
3. **Builder-Breaker** — AI 生成，CY 验证，对抗出质量
4. **每个版本都能跑** — 不做半成品
5. **知识回流** — 踩坑经验录入 Prism（dogfooding）
6. **卡住 2 小时就绕** — 记 pain-log，不死磕
7. **每天 git commit** — 保持可回退
8. **AI 代码先理解再用** — 不做代码搬运工
