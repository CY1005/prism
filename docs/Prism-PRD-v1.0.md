## 1. Problem Statement

### 1.1 问题定义

在 AI 云平台等复杂产品的日常工作中，同一功能模块的知识（产品设计、技术实现、测试分析、竞品对标、工程经验）散落在 5-6 个工具中（Confluence、Notion、Jira、飞书文档、本地笔记、脑子里），导致：

- **信息碎片化**：一个功能的完整画像需要翻 3+ 个工具才能拼出来
- **知识不可追溯**：设计决策、技术取舍的上下文随时间丢失，重复踩坑
- **分析缺乏结构**：需求评审靠经验直觉，缺少系统化的完整性/合理性/影响范围分析框架
- **竞品洞察零散**：竞品信息以碎片形式存在，无法做结构化对标
- **新人/跨团队理解成本高**：没有"一个功能的全貌"视图

### 1.2 谁有这个问题

- 质量工程师（CY 本人）：需要快速建立行业认知、做需求分析和测试点生成
- 产品经理/技术负责人：需要功能全貌视图做决策
- 新入职工程师：需要快速了解系统全景和每个模块的设计意图

### 1.3 证据

- CY 实际工作中维护的知识库文件 200+，分散在多个目录和工具中
- 市场无直接竞品解决"以功能模块为中心的跨维度知识管理"问题
- 现有工具（Confluence/Notion/Jira）均为通用工具，缺乏功能模块中心的组织方式

## 2. Goals & Success Metrics

### 2.1 产品目标

| 目标 | 说明 |
|------|------|
| 行业知识积累 | 系统性建立行业认知体系 |
| 需求分析能力 | 对新需求做完整性、合理性、用户视角的审查 |
| 竞品洞察 | 了解各家产品设计、技术选型、演进方向 |
| 工程经验沉淀 | 每个功能的设计决策和踩坑记录可追溯 |
| 问题归因 | 技术/测试/重构中遇到的问题沉淀到对应功能模块 |

### 2.2 成功指标（OKR 风格）

| 关键结果 | 衡量方式 | 目标值 |
|----------|----------|--------|
| 功能模块录入覆盖率 | 已录入模块数 / 目标产品总模块数 | v0.1: ≥1 个项目完整录入 |
| 知识维度填充率 | 已填维度数 / 启用维度总数 | 平均完善度 ≥60% |
| 需求分析使用频率 | AI 需求分析调用次数 / 月 | v0.2: ≥10 次/月 |
| 冷启动到可用时间 | 从创建项目到录入首个完整模块 | ≤30 分钟 |
| Dogfooding 闭环 | Prism 开发过程的踩坑经验回流到 Prism 自身 | 每个版本 ≥5 条工程经验 |

### 2.3 非目标

- 不使用公司内部数据
- 不做商业化
- 不承担团队工具维护责任
- 不替代现有工具，做分析层叠加

## 3. Target Users / Personas

### 3.1 用户画像

| 角色 | 典型用户 | 核心诉求 | 使用频率 |
|------|----------|----------|----------|
| 质量工程师 | CY（主要用户） | 建立行业认知、需求分析、测试点生成、竞品对标 | 每日 |
| 项目管理员 | 项目负责人 | 掌握项目全貌、追踪功能演进、做技术决策 | 每周 |
| 编辑者 | 团队协作者 | 补充特定维度知识（技术实现/测试分析等） | 按需 |
| 查看者 | 新人/跨团队 | 快速了解系统全景和模块设计意图 | 按需 |

### 3.2 权限模型

| 角色 | 说明 | 权限范围 |
|------|------|----------|
| 平台管理员 | 运维侧 | 用户管理 + 平台配置 |
| 项目管理员 | 项目创建者 | 项目内全部权限 + 成员管理 + AI 配置 |
| 编辑者 | 项目协作者 | 指定项目内的查看 + 编辑 |
| 查看者 | 项目观察者 | 指定项目内的只读 |

**0.x 版本：项目归个人 / 1.x 版本：项目归团队**（迁移成本低，加 team 表即可）

**权限架构预留（v1.x+模块级权限）：**

现阶段权限粒度是项目级，但架构上为未来"角色可见模块不同"提前设计：

| 设计点 | 现在实现 | 预留扩展 |
|--------|---------|---------|
| 权限检查接口 | `checkPermission(userId, projectId, action)` | 接口签名预留 `resourceType` + `resourceId` 参数，现在不传 |
| 数据模型 | `project_members`表（userId, projectId, role） | 预留 `scope` 字段（默认"project"，未来可填"module"/"dimension"） |
| API数据查询 | 返回项目下所有数据 | 所有数据查询经过统一权限过滤层，现在过滤逻辑直通（全返回），未来插入模块级过滤 |
| 前端可见性 | 按角色显示/隐藏操作按钮 | 组件可见性由权限hook驱动（`usePermission()`），不硬编码 |

> **核心原则：现在不实现模块级权限，但确保每个数据出口都经过权限过滤层。** 未来加模块级权限只需改过滤层逻辑，不需要到处加if。

**运维侧：** 用户管理（注册审批、角色分配、禁用）、平台数据统计、AI Provider 全局默认配置

### 3.3 Jobs To Be Done (JTBD)

1. **当我**接到一个新需求时，**我想要**快速查看该功能的全貌（历史设计、技术实现、已知问题、竞品做法），**这样**我能做出更完整的分析而不是从零开始
2. **当我**需要评审需求合理性时，**我想要**AI 基于已有知识库自动分析影响范围和完整性，**这样**我能发现遗漏和风险
3. **当我**积累了大量零散文档时，**我想要**批量导入并自动归类到对应功能模块的对应维度，**这样**知识不再散落
4. **当我**做竞品分析时，**我想要**在同一个功能维度下对比多家实现，**这样**能形成结构化洞察

| 场景 | 预期行为 |
|------|----------|
| 用户创建功能项只填名称，其他全留空 | 允许创建，显示引导文案和完善度进度条 |

## 5. Functional Requirements

### 5.1 MVP（v0.1）-- 能录、能看、能看见结构

#### F1: 用户系统

**描述：** 登录/认证/会话管理。MVP不支持自注册，用户由管理员在运维后台创建。

**MVP AC：**
- AC1: 管理员在运维后台创建用户（输入邮箱+初始密码），创建后用户可登录
- AC2: 已创建用户输入邮箱+密码登录，成功后跳转到/projects；错误时提示"邮箱或密码错误"（不暴露邮箱是否存在）
- AC3: 未登录用户访问任何页面，middleware检查JWT后自动重定向到/login
- AC4: 登录后页面右上角显示用户名，点击可退出登录（清除Cookie）
- AC5: 已禁用账号登录时提示"账号已被禁用，请联系管理员"

**认证方案（方案B）：**
- AC6: 登录成功签发 Access Token（JWT，15分钟过期）+ Refresh Token（7天过期，存DB）
- AC7: Access Token过期后，前端自动用Refresh Token请求新的Access Token，用户无感知
- AC8: Refresh Token续签时检查user.disabled状态，已禁用则拒绝续签并跳转登录页（踢出延迟≤15分钟）
- AC9: 退出登录时清除Cookie + 服务端删除该Refresh Token记录
- AC10: 密码存储使用bcrypt哈希，不明文存储

**管理后台 AC：**
- AC11: 管理后台页面展示用户列表（用户名/角色/状态/创建时间），支持创建用户、禁用用户、修改角色
- AC12: 被禁用用户登录时显示"账号已被禁用"提示，不允许登录

**v1.0预留（MVP只写函数骨架+TODO注释）：**
- TODO: 自注册流程（邮箱+密码注册，注册后自动登录）
- TODO: 密码强度校验规则（长度、复杂度）
- TODO: 密码找回/重置流程
- TODO: OAuth第三方登录（Auth.js已预留扩展点）

**边界场景：**
- 同一用户多设备登录：各设备独立Refresh Token，互不影响
- 管理员禁用账号：已签发的Access Token在15分钟内仍有效，Refresh Token续签时被拦截
- Refresh Token过期（7天未使用）：跳转登录页，需重新输入密码

#### F2: 项目管理

**描述：** 创建/编辑/删除项目，选模板（含维度预设和层级标签），配置 AI Provider，管理成员。创建后跳转引导页引导用户开始填充内容。

**项目创建 AC：**
- AC1: 点击"新建项目"，输入名称+选择模板后创建成功
- AC2: 4个模板可选（产品竞品分析/系统架构/开源项目研究/自定义），每个模板预览显示预设维度列表和层级标签
- AC3: 选择模板创建项目后，自动写入 project_dimension_config（维度启用配置）和 hierarchy_labels（层级标签）
- AC4: 创建成功后跳转**项目引导页**（非全景图），引导页提供三条路径：快速导入（CSV/表格）/ 手动搭建 / AI分析导入。可跳过直接进入项目（详见「项目创建与知识填充流程设计」文档）

**项目列表 AC：**
- AC5: 项目列表页每个卡片显示项目类型Badge，指标标签根据项目类型动态显示（产品线数/组件数/模块数等）
- AC6: 已软删除的项目不在列表中显示

**项目设置 AC：**
- AC7: 项目设置"维度管理"Tab可启用/禁用维度、拖拽排序，保存后档案页维度卡片数量和顺序即时生效。禁用已有数据的维度时提示"数据保留但不显示"
- AC8: 项目设置"层级配置"Tab可修改三层标签名称，保存后树导航和面包屑的文字即时生效
- AC9: 不支持"换模板"——通过AC7加减维度 + AC8改层级标签，可达到任何模板配置的效果

**成员管理 AC：**
- AC10: 项目设置页可邀请成员（输入邮箱+选择角色），被邀请人登录后能看到该项目
- AC11: 查看者只能浏览不能编辑，编辑按钮对查看者隐藏或置灰

**AI Provider 配置 AC：**
- AC12: 运维后台/全局设置管理 API Key 池（增删改，支持多个Provider共存：Claude/Kimi/Codex等）
- AC13: 项目设置中从 Key 池选择默认 Provider，不需要重复填写 Key
- AC14: 每次 AI 操作时（需求分析/竞品对比/智能导入/快照生成）可临时切换 Provider

**项目删除 AC：**
- AC15: 删除项目需二次确认，确认后项目从列表消失（软删除，标记 deleted_at）
- AC16: 软删除的项目关联数据（节点/维度记录/版本记录/关联）保留不动，查询时自动过滤
- AC17: 运维后台可查看已删除项目列表，支持恢复（清除 deleted_at）或彻底删除
- AC18: 软删除超过30天的项目，运维后台可手动彻底删除（级联清理所有关联数据+上传文件）
- AC19: 单条维度记录/节点的删除为级联删除（真删），不走软删除

**模板与项目设置补充 AC：**
- AC20: 创建项目时弹出模板选择器，展示4种模板卡片（产品竞品分析/系统架构/开源项目研究/自定义），选择后预填维度配置
- AC21: 项目设置页"AI配置"Tab，支持配置AI Provider（Claude/Codex/Kimi等），设置默认Provider，API Key加密存储
- AC22: 项目设置页"维度管理"Tab，展示当前启用的维度列表，支持启用/禁用/拖拽排序
- AC23: 项目设置页"层级标签"Tab，支持自定义各层级的显示名称（默认产品线→模块→功能项）

**不支持"换模板"的设计说明：**
> 模板只在创建时起作用（预选一套维度+层级配置）。创建后通过设置里的"加维度""减维度""改层级标签"三个操作，可以调整到任何模板的配置。不需要单独的"换模板"功能，避免已填数据的归属歧义。

#### F3: 功能模块树

**描述：** 左侧展示可配层级标签的树形导航（层级名称由项目配置决定，如"产品线->模块->功能项"或"系统层->组件->功能"）。树不限制层级深度，按功能模块抽象组织（非代码目录结构）。

**树展示 AC：**
- AC1: 进入项目后左侧显示树形结构，层级图标和展开/折叠行为正确
- AC2: 树的层级标签从项目配置的 hierarchy_labels 读取，不同项目显示不同标签
- AC3: 树不限制层级深度（PRD建议3层以内，但不强制限制）
- AC4: 树的每个节点旁显示完善度色块（绿≥80%/黄40-80%/红<40%）
- AC5: 侧边栏顶部显示项目名称和项目类型Badge

**节点选中 AC：**
- AC6: 选中叶子节点，右侧显示对应的功能项档案页
- AC7: 选中文件夹节点，右侧显示该层级的概览页（模块列表+维度统计）

**右键菜单 AC：**
- AC8: 右键点击任意节点，弹出上下文菜单，包含以下操作：
  - 添加子节点：输入名称后新节点出现在该节点下方
  - 重命名：原地编辑节点名称，回车确认，ESC取消
  - 删除：需二次确认（见AC11删除策略）
- AC9: 右键菜单在空白区域点击时，显示"添加顶层节点"选项

**拖拽 AC：**
- AC10: 拖拽节点可调整同级排序
- AC11: 拖拽节点可跨级移动（如把功能项从模块A拖到模块B下面）。跨级移动时自动更新materialized_path整个子树路径

**节点删除 AC：**
- AC12: 删除节点为**级联删除**——该节点及其所有子节点、关联的维度记录、版本记录一并删除
- AC13: 删除前二次确认弹窗，明确提示影响范围："将删除 {节点名} 及其下 {N} 个子节点、{M} 条维度记录，此操作不可撤销"
- AC14: 当前选中节点被删除后，自动切换到父节点；如果删除的是根节点，显示空状态

**拖拽与文件夹概览补充 AC：**
- AC15: 拖拽节点到其他位置时，显示插入指示线和目标高亮，松开后自动更新materialized_path
- AC16: 选中文件夹节点时，右侧展示该文件夹的概要（子节点数量/平均完善度/最近更新），而非空白

**开源项目研究模板的树说明：**
> 即使是从Git导入的开源项目，树结构也按功能模块抽象组织（如"调度系统→Block管理→内存分页"），而非代码目录结构（如"engine/→scheduler/→block_manager.py"）。代码文件路径可以记录在维度内容（代码组织维度）中，但不作为树的组织方式。

#### F4: 功能项档案页

**描述：** 单个功能项的维度一屏展示，核心页面。维度卡片数量和类型根据项目的 dimension_config 动态渲染。内容编辑器使用 Plate（基于Slate.js的shadcn/ui原生富文本编辑器），支持富媒体内容。

**档案页展示 AC：**
- AC1: 点击功能项后右侧展示档案页，维度卡片数量和类型从 project_dimension_config 读取
- AC2: 维度卡片按 project_dimension_config 中的排序顺序渲染
- AC3: 每个维度区块可折叠/展开，折叠时显示摘要（已录入条数或"未录入"）
- AC4: 空维度区块显示引导文案（如"点击添加，或上传文档自动分析"），点击进入编辑模式
- AC5: 面包屑层级标签从项目 hierarchy_labels 读取
- AC6: 创建功能项只需输入名称，其他全部可选
- AC7: 底部版本时间线根据项目的 version_mode 展示不同样式（发布制显示版本号，持续制显示日期）

**完善度计算 AC：**
- AC8: 页面顶部显示完善度进度条，按**字段完整度**计算
- AC9: 计算规则：每条维度记录检查关键字段是否填写，单条完整度 = 已填字段数/该维度定义的总字段数；维度完整度 = 该维度下所有条目完整度的平均值；总完善度 = 各启用维度完整度的平均值（无内容的维度算0%）
- AC10: 完善度示例：竞品参考维度有5个字段（竞品名称/版本/功能覆盖/技术方案/优劣势），一条记录填了3个字段 = 60%；功能描述维度有1个字段（描述文本），填了 = 100%
- AC11: 各维度的"关键字段"定义在 dimension_types.json_schema 中，新增维度类型时必须定义字段清单

**维度内容编辑 AC（Plate编辑器）：**
- AC12: 维度内容编辑器基于 Plate，支持以下内容类型：
  - 文本格式：标题(H1-H3)、加粗、斜体、删除线、引用块
  - 结构：有序/无序列表、表格、分割线
  - 媒体：图片上传/拖拽、文件附件上传
  - 引用：链接、@提及（可以@其他功能项，点击跳转）
  - 代码：行内代码、代码块（带语言选择和语法高亮）
  - 交互：斜杠命令菜单（/image, /code, /table...）、拖拽排序块
- AC13: 编辑器输出JSON格式，直接存入 dimension_records.content (JSONB)
- AC14: 支持Markdown快捷键（输入`# `自动转标题、`**`自动加粗等）

**维度内容添加方式 AC：**
- AC15: "添加"按钮点击后展示4种添加方式：
  - 手动编辑：打开Plate编辑器，富文本编辑
  - 粘贴MD：粘贴Markdown文本，自动转换为Plate结构化内容
  - 上传文件：上传图片/PDF/MD文件，图片作为附件嵌入，MD/PDF由AI解析后填充
  - 从URL导入：粘贴链接，AI抓取摘要后填充
- AC16: 已有内容可编辑（打开Plate编辑器回显内容）和删除（级联删除，真删）

**关联维度引用 AC：**
- AC17: 一条内容可同时归到主维度+关联维度。主维度存原文（source_type="primary"），关联维度存引用（source_type="reference"，指向primary_record_id）
- AC18: 关联维度中的引用内容以**折叠引用**方式展示：默认折叠，显示"▶ 来自「{源维度名}」的关联内容 ({N}条)"，点击展开显示引用内容摘要+[查看原文]跳转链接
- AC19: 引用内容不可在关联维度中直接编辑，编辑需跳转到主维度原文

**多方式录入与关联引用补充 AC：**
- AC20: 维度卡片编辑支持4种录入方式：手动编辑（富文本）、粘贴Markdown（自动渲染）、上传文件（解析内容填入）、URL导入（抓取网页内容）

> **编辑器选型说明：** 暂定Plate，后续评估各组件（Tiptap/Novel/BlockNote）后可调整。核心要求：输出JSON、shadcn/ui风格一致、支持图片上传和代码块。

#### F5: 版本演进时间线

**描述：** 功能项的版本变化历程可视化。版本记录是一个时间标记，关联创建时各维度记录的快照，用于追溯功能在每个版本的状态。

**时间线展示 AC：**
- AC1: 功能项档案页底部展示版本时间线，按时间正序排列
- AC2: 每个时间线节点显示：版本号/日期 + 变更类型图标/颜色 + 摘要 + "当前版本"标签(isCurrent)
- AC3: 产品分析项目显示版本号格式（v3.9.3），系统架构项目显示日期格式（2026-04-07），开源项目研究跟随该项目的release版本号

**版本记录创建 AC：**
- AC4: 点击"添加版本记录"，填写版本号+描述+变更类型后保存，时间线新增节点
- AC5: 保存时自动创建**维度快照**——将该功能项当前所有维度记录的内容做一次快照存储（snapshot_data JSONB），记录该版本时间点的维度状态
- AC6: 版本号不能重复，重复时提示"该版本已存在"

**版本详情 AC：**
- AC7: 点击时间线节点可展开查看该版本的维度快照（只读），展示创建版本时各维度的内容状态
- AC8: 快照内容与当前维度内容可能不同（当前内容可能已被编辑），快照是历史定格

**变更类型 AC：**
- AC9: 变更类型支持6种，用不同颜色/图标区分：
  - 新增（绿色，Plus图标）：全新功能上线
  - 修改（蓝色，Edit图标）：已有功能改动
  - 废弃（灰色，Archive图标）：功能下线/弃用
  - 拆分（橙色，Split图标）：一个功能拆成多个
  - 合并（紫色，Merge图标）：多个功能合成一个
  - 迁移（青色，Move图标）：功能从A模块移到B模块

**边界场景：**
- 无版本记录时：显示"暂无版本记录"
- 拆分/合并时：版本描述中应注明拆分/合并的目标功能项名称，方便追溯
- 维度快照的存储：快照是创建版本时的全量拷贝（非引用），后续维度编辑不影响历史快照

**数据模型：**
```
version_records 表扩展：
  - version: 版本号（唯一约束，同功能项内不重复）
  - description: 版本描述
  - change_type: enum('added', 'modified', 'deprecated', 'split', 'merged', 'migrated')
  - is_current: boolean
  - snapshot_data: JSONB（该功能项所有维度记录的快照）
  - mode: enum('release', 'continuous')
```

### 5.3 v0.3 -- AI分析

#### F13: 需求分析（AI）

**描述：** 基于已沉淀知识，AI分析需求的影响范围、完整性、合理性，并生成测试点

**AC：**
- AC1: 支持粘贴/上传 PRD/MRD 文档（Markdown、Word格式）作为分析输入，不限于纯文字描述
- AC2: AI分析采用渐进式披露，三层上下文逐步加深：
  - **L1 快速分析（默认）**：读取当前功能项的已有维度 + F7已知问题，返回影响范围+完整性+合理性+用户视角分析
  - **L2 关联分析**：L1结果底部显示"扩展分析范围"按钮，追加F8关系图中直接关联模块及其维度作为上下文
  - **L3 全局分析**：L2结果底部显示"全局扫描"按钮，从需求描述提取关键词通过F9搜索筛选相关功能项，将匹配结果喂给AI（不全量灌入，控制token）
- AC3: 每层分析结果标注来源层级（"基于当前功能项""基于关联模块""基于全局扫描"），各层结果保留不覆盖
- AC4: 影响范围在关系图上高亮受影响的模块
- AC5: 分析结果可保存到对应功能项的"需求分析"维度
- AC6: 点击"生成测试点"，AI 基于需求分析结果+已有知识库自动生成测试点列表
- AC7: 生成的测试点可 review 后一键录入对应功能项的"测试分析"维度

> **设计决策：**
> 1. 输入格式支持PRD/MRD文档上传，因为实际需求来源是Markdown/Word文档，不是手打文字。
> 2. 渐进式披露而非固定上下文范围。理由：L1快速+便宜满足80%场景；L2/L3按需加深，用户自己决定分析深度，避免每次都消耗大量token。L3通过搜索筛选而非全量灌入，解决350-400功能项超context window的问题。
> 3. 测试点生成通过Prism后端内部API调用test-workflow Skill，无缝集成，不跳转外部工具。

#### F12: 功能对比矩阵

**AC：**
- AC1: 选择2+个竞品/产品线/版本，AI基于已有知识+联网搜索生成对比表格（竞品从全局竞品实体列表选择）
- AC2: 默认对比维度：功能覆盖度（有/无/部分）、技术方案差异、用户体验差异；支持用户自定义添加对比维度
- AC3: 生成后的对比表支持手动增删改查——可修正AI结果、补充遗漏项、删除无关行
- AC4: 对比表格中差异项自动高亮
- AC5: 对比结果可导出
- AC6: 对比表每行提供"回填"按钮，将该行结论同步到对应功能项的F6竞品参考卡片（批发→零售）

> **设计决策：**
> 1. F12是F6的批量生成器。F6=手动零售录入，F12=AI批发对比后拆条回填。回填粒度是单行（功能项×竞品），不是整张表。
> 2. 对比表由AI生成而非纯数据聚合。理由：350-400个功能项不可能手动填完所有竞品参考，AI批量生成+人工review是唯一可行路径。
> 3. 对比维度可自定义扩展，默认3个维度覆盖最常用场景，用户可按需添加（如性能指标、定价策略等）。

#### F16: AI 生成当前快照

**AC：**
- AC1: 功能项有3+条版本演进记录时，可点击"生成当前快照"
- AC2: AI基于历史版本记录生成两部分：(1) 一句话概要（自由文本，描述当前功能的核心定位）；(2) 按维度结构化输出各维度的最新状态（功能描述/技术实现/用户场景等）
- AC3: 用户review后，一句话概要保存为功能项摘要，结构化部分可按维度选择性覆盖/更新对应维度卡片内容

> **设计决策：** 快照同时输出自由文本+结构化两种形态。一句话概要用于快速理解和对外讲解；结构化输出用于回写维度卡片，形成"版本积累→AI合成→维度更新"的闭环。

#### F15: 数据流转可视化

**描述：** 让用户看见数据在系统里怎么流动的——导入过程实时展示拆分/归类/关联的进度和结果

**AC：**
- AC1: 导入（F11/F17）过程中实时展示：当前处理哪个文件、怎么拆分的、归入了哪个模块、进度百分比
- AC2: 录入/导入完成后，展示流转摘要：导入了N条数据、拆分为M个功能项、关联了K条记录、归入了哪些模块
- AC3: AI分析（F13）完成后，页面提示分析结果流向："分析结果已保存到XX功能项的需求分析维度，生成了N条测试点"
- AC4: 变更日志页展示历史操作列表（时间/操作类型/影响范围），v1.x团队化时扩展"谁"的维度

> **设计决策：** v0.3阶段核心价值是"过程可见"——导入不是黑盒，用户能看到数据怎么被拆分、归类、关联。操作日志做简化版（记录变更列表），"谁"的维度留到v1.x团队化再加（当前单人使用没意义）。

### 5.4 v0.4 -- AI增强+扩展

#### F17: AI 智能导入（知识库批量导入）

**描述：** 在F11基础导入之上，增加AI自动处理能力。支持上传 zip/文件夹，AI 读取文件内容后做结构化处理（拆分、归类、提取、补全、关联、去重、差异标注），用户 review 后批量导入。

> **与F11的关系：** F11=手动映射（用户指定文件归哪个模块），F17=AI自动映射（AI推荐归属+置信度）。F17是F11的AI增强版。

**处理能力：**

| 处理类型 | 说明 | 示例 |
|----------|------|------|
| 拆分 | 一个文件包含多个功能项，拆成独立条目 | `平台功能全景.md` -> 拆成 27 个 RP 的功能描述 |
| 归类 | AI 根据文件内容推荐归属模块和维度 | `故障排查/推理服务OOM.md` -> 模块"推理服务" -> 维度"工程经验" |
| 提取 | 从文件中提取结构化信息填入对应维度 | `双状态机.md` -> 提取状态流转图填入"技术实现"，异常处理填入"测试分析" |
| 补全 | 基于已有维度内容，AI 生成其他维度初稿 | 读完"功能描述"后自动生成"用户场景"初稿 |
| 关联 | 识别文件中的跨模块引用，自动建关联 | 识别"依赖配额链"描述 -> 建立模块间 depends_on 关系 |
| 去重 | 多个文件描述同一功能，合并为不同维度视角 | 运维侧数据管理 + 用户侧数据管理 -> 同一模块的两个视角 |
| 差异标注 | 识别产品线差异描述，自动打标签 | "私有云特有""智算中心不支持" -> 自动标注产品线标签 |

**AC：**
- AC1: 上传 zip 文件后，系统解压并展示文件树，用户可预览每个文件内容
- AC2: 点击"AI 分析"，AI 逐文件读取内容，输出映射表（文件 -> 推荐模块 -> 推荐维度 -> 置信度）
- AC3: 用户可在映射表中调整模块/维度归属，支持批量操作
- AC4: AI 能将单个大文件拆分为多个功能项（如按表格行、按标题层级拆分）
- AC5: AI 能识别文件中的产品线差异描述，自动为功能项打标签（全平台/仅私有云/仅智算中心等）
- AC6: AI 能识别文件中的跨模块引用关系，在确认导入时自动创建模块关联
- AC7: 确认后批量创建功能项和维度记录，展示导入结果摘要（成功/跳过/需人工处理）
- AC8: 导入过程中发现已有同名功能项时，提示合并或跳过，不覆盖已有数据
- AC9: 支持 Markdown（.md）、CSV、纯文本格式的文件解析
- AC10: 导入完成后可一键撤销本次导入（批量删除本次创建的记录）

> **与F15的联动：** F17分两个阶段，对应不同UI：
> 1. **AI分析阶段（确认前）**：F17自己的review映射表——用户逐条审核、调整归属、勾选/排除（交互模式）
> 2. **确认导入后（执行中）**：切换到F15的流转进度面板——实时展示创建进度、拆分结果、关联情况（观察模式）
>
> 不合并两个阶段的UI，因为review和观察进度是两种不同的交互模式——一个要动手改，一个看着就行。

#### F18: Hybrid混合搜索（升级F9）

**描述：** 将F9关键词搜索升级为Hybrid混合搜索，一个搜索框同时走关键词+语义双路召回

**AC：**
- AC1: 搜索框不变（用户无感），后端同时执行BM25关键词搜索+pgvector语义向量搜索
- AC2: 双路结果通过RRF（Reciprocal Rank Fusion）合并排名，两种方法都命中的结果自动加权靠前
- AC3: 搜索"配额"能同时命中包含"quota""额度限制""资源上限"等语义相关结果
- AC4: pgvector不可用时自动降级到纯关键词搜索，不阻塞用户
- AC5: F9已有的全部能力保留（权限控制、路径面包屑、问题覆盖、项目/维度/问题分类筛选）

> **设计决策：** F18不替代也不并存F9，而是升级F9为Hybrid搜索。行业共识：Notion/Obsidian/Confluence都是一个搜索框+后端自动混合，没有产品让用户手动切换"精确/语义"。关键词搜索不能丢——企业用户会搜精确的产品名、工单ID、内部缩写，纯语义搜索反而搜不到。

#### F14: 行业动态（AI推送）

> 从v0.2移至v0.4。理由：优先级低，依赖AI能力就位。

**描述：** AI主动抓取行业动态并推送，自动关联到项目中相关功能项

**AC：**
- AC1: 支持配置RSS/订阅源（AWS Blog、NVIDIA News、竞品官网changelog等），AI定期抓取新内容
- AC2: AI基于项目已有的模块和竞品列表，定期联网搜索相关行业动态
- AC3: 抓取的动态自动结构化（标题/来源/日期/摘要/标签），AI推荐关联的功能项
- AC4: 独立的"动态Feed"页面集中展示待review的动态列表，用户可确认关联/忽略/手动调整关联
- AC5: 确认关联后，动态在对应功能项档案页可见

> **设计决策：** F14不是手动录入，是AI主动推送。来源两路：(1) 用户配置的RSS订阅源定期抓取；(2) AI基于项目知识主动联网搜索。用独立Feed页集中review，而非直接塞进功能项档案页，避免信息噪音——用户决定哪些动态值得关联。

### 5.5 v1.x -- 团队化

#### F19: 导入/导出

**描述：** 支持知识库文档的导入导出，打通Prism与外部知识体系

**AC：**
- AC1: 导入：支持上传Markdown文件/zip压缩包，解析后导入为功能项和维度记录（复用F11/F17的导入能力）
- AC2: 导出：选中模块/功能项后导出为Markdown文件，包含全部维度信息
- AC3: 批量导出：支持按项目/产品线级别导出，生成zip压缩包（目录结构对应模块层级）
- AC4: 导出的Markdown格式可直接被F11/F17重新导入（导入导出格式闭环）

> **设计决策：** 导入复用F11/F17已有能力，不重复建设。导出格式用Markdown而非Word/PDF，因为核心场景是自己知识库的备份和迁移，Markdown可被重新导入形成闭环。

#### F20: 团队/空间

**AC：**
- AC1: 创建团队后可将项目归入团队
- AC2: 现有个人项目可一键迁移到团队
- AC3: 团队成员可查看项目（复用F1已有的管理员/编辑者/查看者权限模型）

> **设计决策：**
> 1. 团队化做最小实现——分组+权限复用F1。不做评论、通知、协作编辑等重协作功能。核心是"万一有人想看"的预留，不是多人协作平台。
> 2. 模块级权限（角色A只能看模块X的维度Y）的架构已在F1权限模型中预留（统一权限过滤层+scope字段+权限hook）。v1.x如果需要，只需扩展过滤层逻辑，不需要重构。

### 8.2 核心数据模型

**架构：** 平台 -> 项目 -> 节点树 -> 叶子节点（功能项 + 维度记录）

| 表 | 职责 |
|----|------|
| project | 项目，含 hierarchy_labels (JSONB)、template_type、version_mode |
| project_dimension_config | 项目x维度启用关系，含排序字段 |
| node | 自引用树（materialized path），created_by/updated_by |
| dimension_type | 维度类型全局注册表（JSON Schema 定义字段），12个预置 |
| dimension_record | 统一存储所有维度数据（JSONB + version 乐观锁） |
| project_template | 项目模板（含预设维度组合和层级标签） |
| node_relation | 节点间关联（不限同项目，支持跨项目引用） |
| version_record | 版本记录，含 mode 字段 (release/continuous) |
| industry_insight | 行业动态 |
| insight_node_link | 动态与节点关联 |

**设计哲学：加新项目类型/分析维度/字段 = 数据操作，不改表结构。**

> 详细数据模型、API 设计、架构图见独立架构文档。

### 8.3 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 维度数据存储 | JSONB 统一表 vs 每维度独立表 | 选 JSONB：扩展性优先，加维度不改表 |
| 树结构实现 | Materialized Path | 查询层级高效，支持拖拽排序 |
| 版本模式 | mode 字段区分 release/continuous | 一套表结构支持两种项目节奏 |
| AI 接入 | 可插拔 LLMProvider 接口 | 不同项目可用不同 AI 后端 |
| 前端架构 | Server Actions 为主 | Next.js 15 原生方案，减少 API 层 |

### 8.4 项目模板与维度系统

**4 种项目模板：**

| 模板 | 层级标签 | 预设维度 | 适用 |
|------|---------|---------|------|
| 产品竞品分析 | 产品线->模块->功能项 | 功能描述、用户场景、技术实现、设计决策、工程经验、测试分析、需求分析、竞品参考 | AI云平台工作知识库 |
| 系统架构项目 | 系统层->组件->功能 | 功能描述、接口规范、设计决策、工程经验、部署配置、测试分析 | OpenClaw |
| 开源项目研究 | 领域->模块->功能 | 项目概述、架构分析、技术亮点、代码组织、技术栈、可借鉴点、问题与局限、学习笔记、同类项目对比 | 从Git导入开源项目做研究学习分析 |
| 自定义 | 用户自定义 | 用户自选 | Prism dogfooding 等 |

> **模板只在创建时起作用**（预选一套维度+层级配置）。创建后通过设置里的"加维度""减维度""改层级标签"可以调整到任何配置，不需要"换模板"功能。

**全局维度类型注册表（18个）：**

核心维度（所有模板共有）：

| 维度 | 说明 | 默认适用模板 |
|------|------|------------|
| 功能描述 | 功能的核心说明 | 产品分析、系统架构 |
| 设计决策 | 关键架构决策及取舍（ADR格式） | 产品分析、系统架构 |
| 工程经验 | 踩坑记录与最佳实践 | 产品分析、系统架构 |
| 测试分析 | 测试策略、问题记录、用例 | 产品分析、系统架构 |

产品分析扩展维度：

| 维度 | 说明 | 默认适用模板 |
|------|------|------------|
| 用户场景 | 谁在什么场景下使用 | 产品分析 |
| 技术实现 | 平台侧的技术方案 | 产品分析 |
| 需求分析 | 需求拆解与影响范围 | 产品分析 |
| 竞品参考 | 竞品功能对标分析 | 产品分析 |

系统架构扩展维度：

| 维度 | 说明 | 默认适用模板 |
|------|------|------------|
| 接口规范 | API接口与协议定义 | 系统架构 |

开源项目研究专属维度：

| 维度 | 说明 | 默认适用模板 |
|------|------|------------|

## 9. Dependencies & Risks

### 9.1 外部依赖

| 依赖 | 说明 | 影响 |

| AI Provider API | Claude Code / Codex / Kimi 等 | AI 分析功能不可用时需优雅降级 |
| test-workflow Skill | CY 已有的需求->测试点能力 | F13 测试点生成依赖此 Skill |
| pgvector 扩展 | v0.3 语义搜索需要 | 仅影响 v0.3，v0.1/v0.2 不依赖 |

### 9.2 风险与缓解

| 风险 | 严重程度 | 缓解方案 |
|------|----------|----------|
| 录入成本高，用户不愿填 | 高 | 只需名称创建；进度条引导；v0.3 AI 辅助 |
| 可视化开发量大 | 中 | React Flow + Recharts，不造轮子 |
| Drizzle ORM AI 熟悉度低 | 中 | 若错误率高切 Prisma，早期迁移成本可控 |
| 做出来不用 | 高 | dogfooding + IDAKE 知识回流 |
| Scope Creep | 中 | AC 约束范围，AC 外的不做 |
| Doom Loop（同一问题反复修不好） | 中 | 同文件编辑>6次停下换思路 |

## 10. Milestones & Sequencing

### 10.1 版本路线图

| 版本 | 功能范围 | 目标 | 构建顺序 |
|------|----------|------|----------|
| **v0.1 MVP** | F1-F5, F11 | 数据进来：能登录、建项目、导入文档、录入维度 | 先交付 |
| **v0.2** | F7, F6, F9, F10, F8 | 看得清：问题沉淀、竞品填充、搜索定位、全景覆盖、依赖关系 | MVP 稳定后 |
| **v0.3** | F13, F12, F16, F15 | AI分析：需求分析、对比矩阵、AI快照、数据流转 | v0.2 数据充足后 |
| **v0.4** | F17, F18, F14 | AI增强：智能导入、语义搜索、行业动态 | v0.3 AI基础就绪后 |
| **v1.x** | F19, F20 | 团队化：导出报告、团队协作 | 个人使用验证后 |

> **重排理由：** 按使用者实际路径排序——先进数据→再组织结构→再看全貌→再AI分析→再增强扩展→最后团队化。原v0.1包含F6/F7（需要先有数据才能填竞品/沉淀问题），移至v0.2；F14行业动态优先级低，移至v0.4；F11导入前置到v0.1（冷启动是第一步）。

### 10.2 v0.1 MVP 建议构建顺序

| 阶段 | 功能 | 依赖 |
|------|------|------|
| Phase 1: 基础骨架 | F1（用户系统）、F2（项目管理） | 无 |
| Phase 2: 核心体验 | F3（模块树）、F4（档案页）、F5（版本时间线） | Phase 1 |
| Phase 3: 冷启动 | F11（zip导入+引导） | Phase 2 |

### 10.3 先交付什么

**最小可用路径：** F1 -> F2 -> F3 -> F4 -> F11

用户能注册登录、创建项目、建模块树、导入已有文档、填写功能项维度。这 5 个功能形成完整的导入-录入-浏览闭环。F11前置确保冷启动可用，不需要手动一条条填350个功能项。
