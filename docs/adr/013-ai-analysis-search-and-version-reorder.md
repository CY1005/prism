# ADR-013: AI分析渐进式披露、搜索Hybrid升级、版本重排与权限架构预留

## Status: Accepted

## Date: 2026-04-11

## Context

F6-F20的需求讨论涉及AI能力设计、搜索策略、功能版本排序、权限架构等跨功能的架构决策。CY的工作强度为满负荷（白天395版本测试 + 晚间Prism开发1-1.5h），版本排序需要基于实际开发节奏。

## Decision

### 决策1：F13 AI分析采用渐进式披露的三层上下文

CY提出不固定选一个上下文层级，而是渐进式加深。

| 层级 | 上下文范围 | 触发方式 | token成本 | 适用场景 |
|------|-----------|---------|----------|---------|
| **L1 快速分析** | 当前功能项的已有维度 + F7已知问题 | 默认，点"分析"直接出 | 低 | "这个需求影响当前功能项的哪些方面" |
| **L2 关联分析** | L1 + F8关系图中直接关联的模块及其维度 | L1结果底部"扩展分析范围"按钮 | 中 | "这个需求会不会影响到依赖模块" |
| **L3 全局分析** | L2 + 项目内所有模块（通过关键词/语义匹配筛选，不全量灌入） | L2结果底部"全局扫描"按钮 | 中高 | "有没有我没想到的影响面" |

**否决的方案：**
- 只读当前功能项（太窄，350-400功能项的跨模块依赖很密）
- 读整个项目知识库（超context window，350-400功能项全量灌入不现实）
- 固定读关联模块（不灵活，有时只需要L1快速分析就够了）

**关键设计：** L3不是全量灌入，而是先用F9搜索能力提取关键词→筛选相关功能项→只把匹配结果喂给AI。每层结果标注来源层级，各层结果保留不覆盖。

**其他F13决策：**
- 输入格式：支持PRD/MRD文档上传（Markdown/Word），因为CY实际需求来源是文档不是手打文字
- test-workflow集成：Prism后端内部API调用，不跳转外部工具（无缝体验 > 实现简单）

### 决策2：F18语义搜索升级F9为Hybrid混合搜索

**调研了行业做法后的决策。**

**讨论过的选项：**

| 方案 | 做法 | 否决/采纳理由 |
|------|------|-------------|
| **A：F18替代F9** | 搜索框统一走语义搜索 | 语义搜索依赖pgvector，挂了连关键词搜索都没了；企业用户搜精确产品名/工单ID时纯语义搜索反而搜不到 |
| **B：F18与F9并存** | 用户切换"精确/语义" | 行业里没有产品让用户手动切换（Notion/Obsidian/Confluence都是一个搜索框） |
| **C：升级F9为Hybrid** ✅ | BM25关键词+pgvector语义双路召回，RRF合并排名 | 行业共识做法，一个搜索框用户无感，pgvector挂了自动降级 |

**行业调研证据：**
- Obsidian VaultSearch插件：BM25 + 语义向量 + RRF合并，两种方法都命中的结果自动加权
- Notion/Confluence：一个搜索框，后端自动融合
- Enterprise Knowledge分析：关键词搜索的可解释性（用户能理解为什么匹配）不能丢
- 企业场景：内部缩写、工单ID等精确匹配需求，embedding模型处理不好

### 决策3：F16 AI快照双输出

**讨论过的选项：**

| 方案 | 做法 | 否决/采纳理由 |
|------|------|-------------|
| **A：自由文本** | 一段综合描述，像功能说明书 | 读起来流畅适合讲解，但无法回写维度卡片 |
| **B：按维度结构化** | 分别输出各维度最新状态 | 可回写维度卡片形成闭环，但缺少概览性 |
| **C：两者都有** ✅ | 一句话概要（自由文本）+ 按维度结构化输出 | CY要求"有个一句话描述" + 结构化也要 |

### 决策4：F14行业动态改为AI主动推送

原PRD设计是手动录入动态。CY明确："这个是要AI自动推送的。"

来源两路：
1. 用户配置的RSS/订阅源（AWS Blog、NVIDIA News、竞品官网changelog）定期抓取
2. AI基于项目已有模块和竞品列表，定期联网搜索

独立Feed页集中review，而非直接塞进功能项档案页——用户决定哪些动态值得关联，避免信息噪音。

### 决策5：版本重排——按使用者实际路径排序

**原排序：** v0.1(F1-F11) → v0.2(F12-F16) → v0.3(F17-F19) → v1.x(F20)

**问题：** F6/F7在v0.1但需要先有数据才能填竞品/沉淀问题；F11冷启动是第一步但排在Phase 5；F17 zip导入放在v0.3太晚。

**新排序逻辑：先进数据→再组织结构→再看全貌→再AI分析→再增强扩展→最后团队化**

| 版本 | 功能 | 定位 |
|------|------|------|
| v0.1 | F1-F5, F11 | 数据进来 |
| v0.2 | F7, F6, F9, F10, F8 | 看得清 |
| v0.3 | F13, F12, F16, F15 | AI分析 |
| v0.4（新增） | F17, F18, F14 | AI增强+扩展 |
| v1.x | F19, F20 | 团队化 |

**关键变更说明：**
- F6/F7从v0.1移至v0.2：先有功能树骨架再填竞品/沉淀问题
- F11从v0.1 Phase 5提到Phase 3：冷启动是第一步
- F14从v0.2移至v0.4：优先级低，不是核心分析能力
- F11改为zip导入（非CSV）：CY的实际数据源是Markdown文档，不是结构化CSV
- F11（手动映射）和F17（AI自动映射）明确分工：F11实现简单v0.1交付，F17是AI增强版v0.4交付

### 决策6：F19导入导出格式闭环

导入复用F11/F17已有能力，导出用Markdown格式。核心场景是自己知识库的备份和迁移，导出的Markdown可被重新导入形成闭环。不做Word/PDF导出。

### 决策7：权限架构预留模块级权限

CY指出：团队化后不同角色可能看到的模块不同，要在架构上提前设计。

**现在不实现模块级权限，但确保每个数据出口都经过权限过滤层：**

| 设计点 | 现在实现 | 预留扩展 |
|--------|---------|---------|
| 权限检查接口 | `checkPermission(userId, projectId, action)` | 预留 `resourceType` + `resourceId` 参数 |
| 数据模型 | `project_members`（userId, projectId, role） | 预留 `scope` 字段（默认"project"，未来"module"/"dimension"） |
| API查询 | 返回项目下所有数据 | 统一权限过滤层，现在直通，未来插入模块级过滤 |
| 前端 | 按角色显示/隐藏按钮 | 权限hook驱动（`usePermission()`），不硬编码 |

核心原则：未来加模块级权限只需改过滤层逻辑，不需要到处加if。

### 决策8：F20团队化最小实现

主要给自己用，团队化只是"万一有人想看"的预留。不做评论、通知、协作编辑等重协作功能。复用F1已有的管理员/编辑者/查看者权限模型。

## Consequences

- **好处**：
  - 渐进式披露让AI分析既快又深，用户自己控制成本
  - Hybrid搜索符合行业共识，不需要用户学习新交互
  - 版本重排匹配实际使用路径，冷启动前置
  - 权限架构预留避免未来重构
- **代价**：
  - 渐进式三层分析增加前端交互复杂度
  - Hybrid搜索依赖pgvector（v0.4才上）
  - 新增v0.4版本，整体交付周期拉长
  - 权限过滤层即使现在直通也要写，增加初期开发量
- **技术债**：
  - test-workflow Skill需要改造为API可调用形式
  - RSS抓取需要后台定时任务基础设施
