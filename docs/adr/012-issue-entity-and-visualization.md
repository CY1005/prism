# ADR-012: 问题独立实体、关系图双级、全景图Treemap与数据流转

## Status: Accepted

## Date: 2026-04-11

## Context

F7（问题沉淀）、F8（关系图）、F10（全景图）、F15（数据流转可视化）的设计需要基于CY实际工作产品的规模来判断——27个模块、350-400个功能项、18,480条测试用例、3条产品线。

## Decision

### 决策1：F7问题是独立实体，按分类关联到对应维度

**讨论过的选项：**

| 方案 | 做法 | 否决/采纳理由 |
|------|------|-------------|
| **A：问题是测试分析维度的子类型** | 所有问题从测试分析入口进 | 简单但语义不精确——技术债、设计缺陷跟测试分析没关系 |
| **B：问题是独立实体，可关联到任意维度** ✅ | 独立 `issues` 表，按分类自动关联 | 技术债→工程经验，设计缺陷→设计决策，bug→测试分析，性能→技术实现 |

**额外确认：**
- 问题定位是**经验沉淀**，不是进度跟踪，不做状态流转（不是Jira）
- F13需求分析时，AI自动读取已知问题作为分析上下文（如"该模块有3条未解决的技术债"）

### 决策2：F8关系图支持模块+功能项两级

**分析过程：** 基于CY工作产品的实际数据规模做判断。

核心论据：
1. 350-400个功能项之间有大量跨模块依赖（推理服务创建→镜像管理→资源组→配额链），这是功能项级的关系，模块级表达不了
2. Bug密集区（配额一致性、调度逻辑、跨视图数据一致性）本质是跨功能项的关联问题
3. 27个模块之间的关系太粗——画出来就是一坨互相连的圆，没有分析价值
4. 3条产品线的差异在功能项级别体现，模块级看不到

**渲染性能控制：** 350-400个功能项全铺开会超PRD的200节点渲染上限。方案是按模块展开/折叠——默认模块级视图（模块折叠为单节点），点击模块展开该模块下的功能项，不做全量铺开。

### 决策3：F10全景图用Treemap，与F8关系图独立Tab

**讨论过的选项：**

| 方案 | 做法 | 否决/采纳理由 |
|------|------|-------------|
| **合并到一个页面做视图切换** | F8和F10共用页面 | 两者解决不同问题，思维模式不同会互相干扰 |
| **两个独立Tab** ✅ | 关系图Tab + 全景图Tab | F8="改了会影响哪里"（链式追踪），F10="填到什么程度了"（全局扫描） |

**Treemap vs 树状展开：**

| | 树状展开 | Treemap ✅ |
|---|---|---|
| 350个节点 | 需要滚动很久，信息密度低 | 一屏看全 |
| "哪里是空的" | 要一行行扫 | 红色色块一眼看到 |
| "哪个模块最重" | 看不出来 | 面积直接体现 |
| 导航 | 强 | 弱，需drill down+面包屑弥补 |

Treemap色块面积=功能项数量，颜色=完善度（红→黄→绿），支持drill down+面包屑导航。

### 决策4：F15数据流转核心是"过程可见"

**讨论过的选项：**

| 方案 | 做法 | 否决/采纳理由 |
|------|------|-------------|
| **偏审计日志** | 记录"谁在什么时候改了什么" | 个人使用阶段"谁"没意义，审计需求留到v1.x团队化 |
| **偏过程可见** ✅ | 导入过程实时展示拆分/归类/关联的进度和结果 | CY的核心需求：导入不是黑盒，要看到数据怎么被处理 |

**与F17的联动设计：** F17（AI智能导入）分两阶段、两种UI：
1. AI分析阶段（确认前）→ F17的review映射表（交互模式：用户要动手调整）
2. 确认导入后（执行中）→ F15的流转进度面板（观察模式：用户看着就行）

不合并两阶段UI，因为review和观察进度是两种不同的交互模式。

## Consequences

- **好处**：问题按维度关联不丢失语义；关系图双级匹配实际分析需求；Treemap信息密度高；数据流转可见增强用户信心
- **代价**：独立issues表增加数据模型复杂度；关系图需要展开/折叠交互；Treemap需要额外可视化库（如d3-treemap或recharts）
