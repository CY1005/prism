# 测试盲区分析

> 本文档记录在手工探索性测试中发现的 bug，分析为什么之前的 E2E 测试没有覆盖到，并提出改进建议。

---

## GAP-001: BUG-086 — 注册页面提交无响应

### Bug 摘要

注册页面点击"注册"后无任何响应。根因：postgres 连接池在 Turbopack HMR 后 stale + handleSubmit 缺少 try-catch 导致错误被静默吞掉。

### 为什么 E2E 没测出来

**1. 测试点设计盲区：155 个测试点中没有"用户自注册"相关测试**

F1 用户系统的 14 个测试点覆盖了：
- 登录/Token/受保护端点（TP-001, TP-002）
- 错误密码/不存在邮箱（TP-021, TP-022）
- /api/auth/me（TP-023）
- Refresh Token 续签/退出登录（TP-024, TP-025）
- 密码哈希存储（TP-026）
- 管理员创建/禁用用户（TP-027, TP-028）
- 账号锁定/JWT吊销等回归测试

**完全缺失**：用户自助注册流程（`/register` 页面 → Server Action → 自动登录 → 跳转）

**2. 测试层级盲区：所有 F1 测试点都是 API 层，没有前端层**

所有测试步骤格式为 `curl -X POST http://127.0.0.1:8001/api/auth/...`，直接调用 FastAPI 后端。这意味着：
- 不经过 Next.js Server Action 层
- 不经过前端表单 + React Transition
- 不经过 Turbopack 模块打包
- 不经过 postgres.js 在 Next.js 环境下的连接管理

这个 bug 恰好只在 **Next.js Server Action 环境** 中触发，纯 API 调用不会暴露。

**3. 测试基础设施盲区：使用种子用户，绕过了注册流程**

E2E 测试前置条件为"种子用户 cy@prism.dev 存在"，所有后续测试直接用该用户登录。管理员创建用户走的是 API 接口 `POST /api/auth/users`，不走前端注册页面。

### 改进建议

| 编号 | 改进项 | 优先级 |
|------|--------|--------|
| 1 | 补充 `/register` 页面的 E2E 测试点（表单提交 → 注册成功 → 自动登录 → 跳转） | P0 |
| 2 | 补充 `/login` 页面的前端层测试点（Server Action 路径，不只是 API 路径） | P1 |
| 3 | 对于有前端页面的功能，测试点应覆盖"前端 → Server Action → 后端"全链路，不能只测 API 层 | P1 |
| 4 | 测试环境启动后，应验证 Server Action 可正常调用 DB（作为冒烟测试的一部分） | P2 |

### 根本反思

> **测试点基于 PRD Feature 设计，但 PRD 里"用户系统"只描述了认证/授权能力，没有明确列出"用户自注册"作为独立验收条件。测试跟着 PRD 走，PRD 没写的就没测。**
>
> 这暴露了一个系统性问题：当 VibeCoding 生成了 `/register` 页面，但 PRD 没有对应 AC 时，测试设计不会自动覆盖到。改进方向是在测试点设计阶段，除了对齐 PRD AC，还应扫描所有可访问的前端路由，确保每个页面至少有一个端到端测试点。

## GAP-002: BUG-087 — Button asChild 失效导致 icon 文案串行

### Bug 摘要

全项目 22 处 `<Button asChild><Link>` 的 icon 和文案垂直串行。根因：Button 组件迁移到 base-ui 时丢弃了 `asChild` prop。

### 为什么 E2E 没测出来

**1. E2E 测试全部是 API 层，没有 UI 层验证**

155 个测试点的验证方式全部是 `curl` + JSON 响应检查。没有任何测试点验证前端页面的 DOM 结构或视觉呈现。

**2. "icon 和文案对齐"属于视觉回归，不在功能测试范畴**

即使有前端测试，功能性 E2E 通常验证"按钮是否存在、点击是否跳转"，不会验证"icon 和文案是否在同一行"。这类问题属于视觉回归测试范畴。

**3. 组件库迁移是全局影响，但测试按 Feature 隔离**

`asChild` 失效影响全项目 22 处按钮，但按 Feature 划分的测试点不会交叉验证共享组件的行为变化。

### 改进建议

| 编号 | 改进项 | 优先级 |
|------|--------|--------|
| 1 | 组件库迁移后增加一轮 UI 冒烟（打开每个页面截图对比） | P1 |
| 2 | 关键页面增加 DOM 结构断言（如 Button 内子元素是否在 flex 容器内） | P2 |
| 3 | 建立"全局组件变更 → 全页面视觉回归"的检查清单 | P2 |

### 根本反思

> **API-only 的 E2E 测试天然不覆盖前端渲染问题。** 这不是测试点设计的遗漏，而是测试层级的结构性盲区。当项目同时有 API 和前端时，仅有 API 层 E2E 是不够的——至少需要对关键页面做一次"打开能看"的冒烟验证。
>
> 更深层的问题是：**VibeCoding 迭代中的组件库迁移（Radix → base-ui）没有被当作"破坏性变更"对待。** 共享组件的 API 变化应该触发全面的 UI 回归，而不是默默标注 "unused" 就完事。

## GAP-003: BUG-088/089 — 导入页面文案矛盾 + 重复标题

### Bug 摘要

- BUG-088：上传区域文案前后矛盾（上面说 ZIP，下面说 Markdown），用户不清楚到底支持什么
- BUG-089：导入页面有两个"导入文档"标题头

### 为什么 E2E 没测出来

**1. 文案验证不在 E2E 测试范畴**

155 个测试点关注的是"功能是否可用"（上传成功/失败、映射是否正确），不会验证上传区域的引导文案是否准确、是否自洽。

**2. 组件嵌套的集成状态未测试**

ImportWizard 有 `standalone` 参数控制是否渲染自己的 header。但 E2E 测试只验证"导入是否成功"，不验证页面结构。组件在独立使用和嵌套使用时的 UI 差异没有覆盖。

**3. 测试点缺少 UX 维度的验证**

功能测试本身就包含 UX 测试。"文案是否自洽"、"页面结构是否合理"都是功能测试应该覆盖的范围——用户看到矛盾的引导文案无法正确操作，这就是功能缺陷，不是独立于功能之外的"体验问题"。之前的测试点只验证了"上传是否成功"，没有验证"用户能否根据页面引导理解该做什么"，这是测试点设计的遗漏。

### 改进建议

| 编号 | 改进项 | 优先级 |
|------|--------|--------|
| 1 | 功能测试点应包含 UX 维度：页面引导文案与实际行为是否一致、信息层级是否清晰 | P1 |
| 2 | 有 standalone 参数的组件，测试点应覆盖两种模式（独立 + 嵌套），验证页面结构不冗余 | P1 |
| 3 | 每个功能的测试点增加"首次使用者视角"检查：仅凭页面信息，用户能否完成操作 | P1 |

### 根本反思

> **功能测试 = 功能可用 + 用户可理解。** 如果页面引导文案让用户无法判断该上传什么文件，那这个功能就是不可用的——即使后端逻辑完全正确。之前的测试点把"功能"窄化为"API 调通"，忽略了用户交互层面的可用性。测试点设计应该从用户操作流程出发，而不仅仅从接口出发。

## GAP-004: BUG-090 — 上传 ZIP 报 "fail to fetch"

### Bug 摘要

上传 ZIP 文件时 Server Action body 超过 Next.js 默认 1MB 限制，请求被拒绝。

### 为什么 E2E 没测出来

**1. E2E 测试的上传走的是 FastAPI 后端 API，不经过 Next.js Server Action**

之前的导入测试点（如果有）直接调用 `POST /api/import/upload` 到 FastAPI 端口 8001，文件大小限制由 FastAPI 控制（默认无限制或很大）。而真实用户走的是前端页面 → Server Action → FastAPI，中间多了 Next.js 这一层，它有自己的 1MB body 限制。

**2. 测试用例使用的是小文件**

测试 fixtures 目录下的 `test-import.zip` 和 `edge-case-import.zip` 很可能小于 1MB，即使走了 Server Action 也不会触发限制。真实用户的文档包通常远大于测试数据。

**3. 框架默认配置没有被当作测试点**

"文件大小限制"是一个跨层关注点——前端文案说 50MB、Server Action 默认 1MB、FastAPI 可能又是另一个值。但测试点只验证"上传成功/失败"，没有验证"上传 N MB 的文件是否成功"。

### 改进建议

| 编号 | 改进项 | 优先级 |
|------|--------|--------|
| 1 | 文件上传功能的测试点必须包含边界值：0 字节、1MB 附近、文案标注的上限 | P0 |
| 2 | 端到端测试应走真实用户路径（前端 → Server Action → 后端），而非直接调后端 API | P1 |
| 3 | 框架默认配置（body 限制、超时、并发数）应作为基础设施测试点纳入冒烟测试 | P1 |

## GAP-005: BUG-091/093 — 导入预览无法滚动 / 左右面板滚动干扰

### Bug 摘要

上传 ZIP 进入预览步骤后，文件内容超出可视区域但没有滚动条。左右面板滚动互相干扰。

### 为什么 E2E 没测出来

**1. E2E 测试不验证页面交互状态**

155 个测试点验证的是"API 返回正确数据"，不验证"页面能不能滚动"。滚动是前端渲染+CSS 布局的问题，纯 API 测试无法覆盖。

**2. 测试数据量不足以触发滚动**

测试 fixtures 里的 ZIP 文件可能只有几个小文件，内容不足以超出一屏，即使手工测也不会注意到"滚不动"。

**3. 组件嵌套模式变更无回归**

ImportWizard 有 standalone 和嵌套两种模式。嵌套模式下多了一层外壳（ImportPageClient），改变了 CSS 高度约束链路。没有针对"同一组件在不同嵌套深度下的布局表现"的测试。

### 改进建议

| 编号 | 改进项 | 优先级 |
|------|--------|--------|
| 1 | 涉及滚动的页面，测试数据必须超过一屏高度 | P1 |
| 2 | 组件有 standalone/嵌套两种模式时，两种模式都要测试完整交互流程 | P1 |
| 3 | CSS 布局方案选型时优先用 absolute 定位而非深层 flex 链条，降低回归风险 | P2 |

## GAP-006: BUG-094 — "use server" 导出 object 导致页面崩溃

### Bug 摘要

issues.ts 导出了一个 const object，Next.js 16 严格模式下 "use server" 文件只允许导出 async function，整个页面崩溃。

### 为什么 E2E 没测出来

**1. E2E 测试在 Next.js 14/15 环境下执行**

之前的 E2E 测试是在旧版 Next.js 下跑的，旧版不检查 "use server" 的导出类型。升级到 Next.js 16 后这个检查变严格了，但没有重跑 E2E。

**2. 没有"框架升级后的兼容性冒烟测试"**

框架大版本升级（14→16）是破坏性变更，应该有专门的兼容性检查清单。"use server" 文件的导出规则变化属于 Next.js 16 的 breaking change，但升级时没有逐个文件检查。

### 改进建议

| 编号 | 改进项 | 优先级 |
|------|--------|--------|
| 1 | 框架大版本升级后，必须完整重跑一遍 E2E + 打开每个页面验证无报错 | P0 |
| 2 | CI 中增加 `next build` 检查，编译时就能发现 "use server" 导出问题 | P0 |
| 3 | "use server" 文件的 lint 规则：禁止导出非 async function | P1 |

## GAP-007: BUG-095 — AI 导入报"未登录"，服务间认证缺失

### Bug 摘要

AI 导入分析时，Next.js server action 调用 FastAPI 没传认证 token，FastAPI 返回 401"未登录"。

### 为什么 E2E 没测出来

**1. E2E 测试直接调 FastAPI 并手动传 Bearer token**

测试步骤是 `curl -H "Authorization: Bearer $TOKEN" http://localhost:8001/api/import/ai-analyze`。这绕过了 Next.js server action 层，直接跟 FastAPI 对话。Token 是手动获取并传递的，所以 FastAPI 认证通过。

**但真实用户走的是**：浏览器 → Next.js server action（有 session）→ FastAPI（没 token）→ 401

**2. 两套认证体系的桥接从未被测试**

Next.js 用 Auth.js cookie session，FastAPI 用 Bearer JWT token。之前的测试分别验证了两套体系各自能工作，但从未验证"Next.js 调 FastAPI 时认证是否打通"。

**3. 这是典型的"集成缝隙"**

单元/功能测试各自通过，但系统集成点没有覆盖。server action 里的 `requireAuth()` 通过了≠FastAPI 的 `require_user` 也通过了。

### 改进建议

| 编号 | 改进项 | 优先级 |
|------|--------|--------|
| 1 | **所有需要认证的功能，必须从前端页面发起测试**，不能只测后端 API | P0 |
| 2 | 增加"服务间认证"冒烟测试：从 server action 调 FastAPI 需认证端点，验证 200 | P0 |
| 3 | 架构文档标注哪些 FastAPI 端点需要认证，server action 调用时必须带内部 token | P1 |

### 根本反思

> **这是本次测试发现的最关键的系统性盲区：E2E 测试全部直接调 FastAPI，绕过了 Next.js server action 层。** 而真实用户 100% 走 server action 路径。这意味着 155 个测试点验证的是"FastAPI 能不能工作"，而不是"用户的操作能不能成功"。两者之间隔了一个 server action 层，这一层有自己的认证、错误处理、数据转换，这些都是未测试的。

## GAP-009: BUG-097/098 — DeepSeek 不在下拉 + 保存失败静默

### Bug 摘要

- 前端下拉和后端校验都没有 DeepSeek 选项，但后端 provider 实现支持
- 保存失败时前端没有任何错误提示

### 为什么 E2E 没测出来

**1. Provider 列表一致性从未被测试**

E2E 测试只测了"已配置好的项目能不能用 AI"，没有测试"在设置页面配置一个新 provider 并保存"。三处 provider 列表（后端实现、server action 校验、前端下拉）的一致性没有测试点覆盖。

**2. "保存失败"不在正向测试路径上**

测试点验证的是"保存成功后配置生效"。没有反向测试"保存一个无效的 provider 会怎样"。更关键的是，由于 BUG-098 的存在，即使保存失败页面也没有反馈，手工测试时也很难注意到——你以为成功了，实际没保存。

**3. 这是"配置一致性"的系统性问题**

三处维护同一份列表（后端 `provider_map`、server action `validProviders`、前端 `SelectItem`），没有单一真相源。新增 provider 时靠人记住改三个地方，必然遗漏。

### 改进建议

| 编号 | 改进项 | 优先级 |
|------|--------|--------|
| 1 | Provider 列表提取为单一真相源（shared config），三处引用同一份 | P0 |
| 2 | 设置页面的测试点应包含"修改配置 → 保存 → 刷新 → 验证持久化"的完整链路 | P0 |
| 3 | 所有 server action 调用方检查返回值——这是第三次发现同类问题，需要建立编码规范 | P0 |

### 根本反思

> **"server action 返回值未检查"已经是本轮测试的第一高频问题类型**（BUG-086 注册、BUG-098 设置、以及 workspace 里多处 handleConfirmAddNode 等）。这不是个别遗漏，而是 VibeCoding 的系统性编码模式缺陷——AI 生成的代码倾向于写 `await action(); doNext()`，不写 `const result = await action(); if (!result.success) handleError()`。需要作为 Code Quality Gate 的检查规则。

## GAP-010: BUG-099 — 保存成功后无视觉反馈

### Bug 摘要

AI 配置保存成功后，API Key 输入框清空，无任何成功提示。加密存储不回显原文是正确的安全设计，但缺少"已配置"状态指示。

### 为什么 E2E 没测出来

**1. E2E 测试验证的是"数据持久化"，不是"页面反馈"**

测试步骤是：保存配置 → 查询数据库 → 验证值正确。只要数据库写入成功就 PASS。不验证"用户保存后能否在页面上确认保存成功了"。

**2. 加密字段的 UX 是特殊场景**

API Key 加密后不回显是正确的安全设计，但这让"保存成功"和"未配置"在视觉上无法区分。E2E 测试不覆盖这种"两种不同状态看起来一样"的 UX 问题。

### 改进建议

| 编号 | 改进项 | 优先级 |
|------|--------|--------|
| 1 | 涉及加密字段的配置页面，测试点必须包含"保存后刷新，验证页面能区分已配置/未配置" | P1 |
| 2 | 所有表单保存操作的测试点应包含"成功后页面给出明确反馈" | P1 |
