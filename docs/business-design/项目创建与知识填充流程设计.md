# Prism — 项目创建与知识填充流程设计

> 解决的问题：PRD对"创建项目后怎么用起来"描述不清晰
> 涉及功能：F2项目管理 / F4档案页 / F11冷启动 / F17智能导入 / 维度富媒体支持
> 状态：设计稿，待确认后回写PRD

---

## 一、现有PRD的问题

| 问题 | 现状 | 影响 |
|------|------|------|
| 创建后无引导 | F11只写了"引导用户选一个模块先填完"，没有具体交互 | 用户创建完项目面对空白页不知道下一步做什么 |
| 导入路径不清晰 | CSV导入(F11)和AI智能导入(F17)分属不同版本，入口分散 | 用户不知道什么时候用哪个导入 |
| 维度内容只支持文本 | PRD没有提到图片/链接/代码块支持 | 实际使用中截图、API文档链接、代码片段很常见 |
| AI分析产出无归位 | AI分析了技术文档产出MD，但没有"把分析结果拆到对应维度"的流程 | 用户手动复制粘贴，失去AI辅助的意义 |

### 3.3 三条路径详情

**路径A：快速导入（有现成清单）**
- 适用：已有功能模块清单（Excel/CSV/飞书表格复制）
- 流程：上传CSV或粘贴表格 → 预览解析（列映射） → 确认 → 批量创建模块树骨架
- 产出：模块树结构搭好（只有名称），维度内容为空，引导用户去填充

**路径B：手动搭建（从零开始）**
- 适用：没有现成文档，想边想边建
- 流程：引导创建第一个产品线/系统层 → 引导创建第一个模块 → 引导创建第一个功能项 → 引导填写"功能描述"维度
- 产出：一个最小完整路径（1个功能项填完1个维度），展示全景图（Aha Moment）

**路径C：AI分析导入（有技术文档）**
- 适用：有Markdown文档、技术分析、竞品报告等（AI产出的或自己写的）
- 流程：见下方第四节

## 四、Phase 2：AI分析导入流程（核心创新点）

### 4.1 场景描述

用户有一份AI帮忙分析的技术文档（如Gemini分析了某产品的API文档，输出了一份MD），想把这份文档的内容**自动拆解到Prism的功能模块和维度里**。

### 4.2 流程设计

```

│ · 粘贴文本     │            │ · 推荐归属模块 │            │ · 拆分/合并    │          │ · 完善度更新   │

### 4.4 AI归位的判断依据

> **设计教训**：知识库迁移时曾反复犯"关键词匹配分类"的错——看到"配置"就放部署与配置、看到"GPU"就放集群与GPU。根因是把分类当成文件搬迁而非知识重组。维度匹配必须避免同样的错误。

AI怎么知道一段文字该归到哪个模块、哪个维度？**不能靠关键词特征，必须靠上下文理解 + 用户意图。**

#### 4.4.1 模块匹配（上下文感知）

**第一步：读已有知识**
- AI先读取项目中已有的模块树和各模块已填充的维度内容，建立项目的知识结构心智模型
- 匹配时基于语义相似度，而非关键词命中

**第二步：语义匹配**
- 如果文档提到"副本调度的乐观锁机制"，不是匹配"副本"这个词，而是理解这段内容在讨论推理服务的资源调度，匹配到模块"创建推理服务"或"自动扩缩容"
- 如果匹配不到已有模块，AI建议创建新模块（用户确认）

**第三步：相似内容参考**
- 匹配时展示该模块已有的相关内容摘要，帮用户判断是否归对了
- 例："推荐归到「创建推理服务」，该模块已有：功能描述(1条)、技术实现(2条)、工程经验(1条)"

#### 4.4.2 维度匹配（意图驱动，非关键词驱动）

~~旧方案（已废弃）：基于内容特征词匹配——含"技术方案"→技术实现，含"问题"→工程经验~~

**为什么关键词匹配不行：**

真实知识库中的反例：

| 真实文档内容 | 关键词会归到 | 实际应归到 | 为什么错 |
|------------|-----------|---------|---------|
| "采用Control Loop模式，声明式而非命令式"（副本调度架构分析） | 技术实现（含"模式""架构"） | **设计决策** | 这段在论证"为什么选这个模式"，是取舍不是实现 |
| "当前平台已配置Bearer鉴权但零限流策略部署"（API-Key鉴权分析） | 工程经验（含"问题"） | **测试分析** | 这是需要验证的现状gap，不是踩坑记录 |
| "四种限流算法：固定窗口/滑动窗口/令牌桶/漏桶"（同一份文档） | 技术实现（含"算法"） | **取决于用户意图** | 学习用→技术实现；选型→设计决策；做测试→测试分析 |
| "扩缩容在非运行中状态下不生效，需验证边界"（基础概念） | 工程经验（含"踩坑"）| **同时属于**技术实现+测试分析+工程经验 | 强行归一个维度会打碎知识关联 |

**新方案：三层判断机制**

```
Layer 1: 上下文感知（AI先读项目已有知识）
  ↓ 理解这个项目的知识结构和各维度已有内容
  
Layer 2: 意图询问（问用户"这段内容你拿来做什么"）
  ↓ 用户选择：学习理解 / 做决策 / 做测试 / 记录经验 / 对标竞品
  
Layer 3: 多维度归位（一条内容可归多个维度）
  ↓ 主维度 + 关联维度，引用关系而非复制
```

**Layer 1：上下文感知匹配**

AI在判断维度前，先读取目标模块已有的维度内容：

```
目标模块："创建推理服务"
已有内容：
  - 功能描述(1条): "推理服务支持多模型部署..."
  - 技术实现(2条): "使用vLLM作为推理引擎..." / "采用K8s Deployment管理副本..."
  - 设计决策(1条): "选择声明式API而非命令式..."
  - 工程经验(0条): 空
  - 测试分析(0条): 空

新内容："副本调度采用Control Loop模式，双字段状态管理(current+target)，
        乐观锁防并发冲突。高风险：双入口并发修改时后写覆盖先写。"

AI判断：
  - 这段内容与已有的"技术实现-K8s Deployment"和"设计决策-声明式API"相关
  - "Control Loop模式+双字段状态"是架构选择的论证 → 主维度：设计决策
  - "乐观锁防并发冲突"是具体实现细节 → 关联维度：技术实现
  - "高风险：双入口并发"是需要验证的场景 → 关联维度：测试分析
```

**Layer 2：意图询问**

当AI无法确定主维度时（置信度<70%），不猜测，而是问用户：

```
┌─────────────────────────────────────────────────────────┐
│ 这段内容可以从多个角度使用，你打算拿它来：               │
│                                                         │
│  ○ 理解这个功能怎么实现的（→ 技术实现）                  │
│  ○ 记录设计取舍和理由（→ 设计决策）                      │
│  ○ 作为需要验证的测试场景（→ 测试分析）                  │
│  ○ 记录踩坑教训（→ 工程经验）                           │
│  ○ 对标竞品做法（→ 竞品参考）                           │
│  ○ 以上都有，帮我拆开（→ AI拆段后分别归位）              │
└─────────────────────────────────────────────────────────┘
```

**Layer 3：多维度归位**

一条知识片段可以同时关联到多个维度，不强制1:1：

```
知识片段："副本调度采用Control Loop + 乐观锁 + 双入口并发风险"
  │
  ├── 主维度：设计决策（"为什么选Control Loop"）
  │     └── 内容：完整原文
  │
  ├── 关联维度：技术实现（"乐观锁怎么实现的"）
  │     └── 内容：引用 → 点击跳转到主维度原文
  │
  └── 关联维度：测试分析（"双入口并发需要验证"）
        └── 内容：引用 → 点击跳转到主维度原文
```

**数据模型支持**：

```json
// dimension_records.content 扩展
{
  "text": "副本调度采用Control Loop模式...",
  "source_type": "primary",  // "primary" = 主维度原文, "reference" = 关联引用
  "primary_record_id": null, // 主维度时为null，引用时指向主记录ID
  "related_dimensions": ["技术实现", "测试分析"]  // 主维度时列出关联维度
}
```

### 4.5 如果模块树还是空的

首次导入时模块树是空的，AI需要两步走：
1. **先从文档中提取模块结构**（建骨架）
2. **再读取刚建好的骨架作为上下文，把内容片段归到对应模块+维度**

```
Step 0 (仅首次): AI从文档中提取模块树结构
  "这份文档涉及以下模块，建议创建："
  ├── 推理服务
  │   ├── 创建推理服务
  │   ├── 路由管理
  │   └── 自动扩缩容
  ├── 基础设施
  │   ├── GPU管理
  │   └── 存储管理
  └── [用户确认/调整后创建]

Step 1: 用户确认模块树后，AI以此为上下文执行归位
Step 2-4: 同上正常流程（上下文感知 + 意图询问 + 多维度归位）
```

## 五、Phase 3：维度内容富媒体支持

### 5.1 当前问题

PRD中维度内容只定义了文本结构（JSONB），没有提到图片、链接、附件。实际使用中需要：

- **截图**：蓝湖截图、竞品截图、架构图
- **链接**：API文档URL、Confluence链接、GitHub链接
- **代码块**：配置示例、API调用示例
- **附件**：PDF技术文档、设计稿

### 5.2 设计方案

维度卡片的内容编辑器支持以下富媒体类型：

```

### 5.3 数据模型扩展

现有 `dimension_records.content` 是 JSONB，扩展为支持富媒体：

```json
{
  "text": "推理服务支持多模型部署...",
  "attachments": [
    { "type": "image", "name": "架构图.png", "url": "/uploads/xxx.png", "size": 245000 },
    { "type": "file", "name": "API文档.pdf", "url": "/uploads/xxx.pdf", "size": 1200000 }
  ],
  "links": [
    { "label": "蓝湖设计稿", "url": "https://lanhu.io/xxx" },
    { "label": "API文档", "url": "https://docs.xxx/inference" }
  ],
  "codeBlocks": [
    { "language": "python", "code": "client.create_inference(...)" }
  ]
}
```

不需要改表结构（JSONB的优势），只需要前端渲染器和编辑器支持。

## 七、入口整合

所有导入/创建/AI分析的入口应该收敛到**两个统一入口**：

### 入口1：项目级 — 导入数据（批量）
位置：全景图右上角"导入数据"按钮

```
导入数据
├── 快速导入（CSV/表格粘贴）     → 创建模块树骨架
├── AI智能导入（MD/PDF/zip上传）  → AI拆解 + 归位
└── URL导入（粘贴文档链接）       → AI抓取 + 拆解 + 归位
```

### 入口2：功能项级 — 添加内容（单个）
位置：维度卡片的"添加"按钮

```
添加内容
├── 手动编辑（富文本+附件+链接+代码块）
├── 粘贴MD（AI自动识别归哪个维度）
├── 上传文件（图片/PDF → AI分析后归位）
└── 从URL导入（粘贴链接 → AI抓取摘要）
```
