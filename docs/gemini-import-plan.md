## 六、导入策略

### 阶段一：骨架导入（CSV批量）

1. 先导入第1层（4条产品线）和第2层（26个模块）
2. 每个模块的"功能描述"维度从 `_全景/平台功能全景-R3.9.3.md` 提取
3. 建立模块间的关联关系（如 GPU管理 → 节点管理，推理服务 → 任务提交通用能力）

### 阶段二：维度填充（按模块逐个）

按知识库目录逐个模块填充维度内容：

| 维度 | 来源 |
|------|------|
| 功能描述 | `平台功能全景-R3.9.3.md` 中的功能点表格 |
| 用户场景 | 各模块 `_overview.md` 中的使用场景 |
| 技术实现 | `_组件/` 目录下的组件分析文件 |
| 设计决策 | `04-项目/Prism/架构决策总结.md` + 各模块特有决策 |
| 工程经验 | `排查与运维/故障排查/` 中的踩坑记录 |
| 测试分析 | `03-测试/` 中对应的测试策略和用例 |
| 需求分析 | `版本工作台/395/` 中的当前版本需求 |
| 竞品参考 | `_行业/` 目录下的竞品分析 |

### 阶段三：差异点标注

利用 `私有云 vs 智算中心差异点汇总` 表，为差异功能项打标签：
- `全平台` / `仅私有云` / `仅智算中心` / `仅公有云`

### 阶段四：关系网络

基于 `平台架构导航地图.md` 中的架构图和数据流转链路，建立模块关联：
- 资源链：集群 → 节点 → GPU → 资源组 → 空间配额 → 任务
- 训练链：项目 → 代码/数据/镜像 → 开发环境 → 训练 → 结果 → 模型
- 推理链：推理服务 → 路由 → 扩缩容 → 版本管理 → 监控
- 调度链：任务提交 → 配额检查 → 排队 → 调度 → 启动
